# 🔥 Claude Code Ops 4.6 分析。高推理

当 Anthropic 发布像 Opus 4.6 这样的模型时，**模型权重**——数十亿学习参数——是冻结的。训练成本极高（数百万美元、数周计算）。没有人会在一夜之间重新训练模型。

但权重只是更大系统的其中一层。研究显示，即使模型权重冻结，也至少有**7 种不同的机制**会导致真实或感知到的质量变化。

| 问题 | 答案 |
|----------|--------|
| 模型权重在发布后会改变吗？ | **否** — 所有提供商确认 |
| 模型逐日表现可能不同吗？ | **是** — 已证实 ±8-14% 差异 |
| 是故意的"削弱"吗？ | **否** — 无故意退化证据 |
| 基础设施 bug 是真实的吗？ | **是** — Anthropic 确认 3 个 bug 影响高达 16% 的请求 |
| 部分是心理因素吗？ | **是** — 确认偏差和蜜月效应是真实的 |
| 系统提示/后训练可以改变吗？ | **是** — 各提供商有文档记录 |
| 用户应该相信他们的感知吗？ | **部分** — 真实原因存在，但感知会放大它们 |

## 完整推理栈

模型权重是冻结的，但**在其之上的九层**可以独立影响你的体验：

```
┌──────────────────────────────────────────────┐
│  你的会话上下文                        │  ← 会话内退化
│  （累积的错误、长对话）                   │
├──────────────────────────────────────────────┤
│  系统提示                            │  ← 定期更新
│  （安全规则、行为指令）                   │
├──────────────────────────────────────────────┤
│  后训练（RLHF / 微调）               │  ← 可以静默更新
│  （指令遵循、安全对齐）                   │
├──────────────────────────────────────────────┤
│  采样参数                           │  ← 可在服务器端调整
│  （temperature、top-p、top-k）             │
├──────────────────────────────────────────────┤
│  投机解码                           │  ← 草稿模型质量各异
│  （草稿模型预测 + 验证）                    │
├──────────────────────────────────────────────┤
│  MoE 路由 / 批处理组成                │  �� ±8-14% 差异已证实
│  （每个请求激活哪些专家）                   │
├──────────────────────────────────────────────┤
│  硬件路由                           │  ← TPU vs GPU vs Trainium
│  （哪个集群服务你的请求）                  │
├──────────────────────────────────────────────┤
│  量化级别                           │  ← 负载下可能变化
│  （FP16 vs INT8 vs INT4 精度）            │
├──────────────────────────────────────────────┤
│  编译器和运行时                       ←  ← XLA bug 已证实真实
│  （XLA:TPU、CUDA、硬件特定代码）          │
├──────────────────────────────────────────────┤
│  模型权重（冻结）                     │  ← 这些不变化
│  （数十亿学习参数）                    │
└──────────────────────────────────────────────┘
```

关键心智模型：**冻结的权重 ≠ 冻结的行为**。这就像说"相同的引擎 = 相同的驾驶体验"，同时忽略了轮胎、道路状况、燃料质量和驾驶员疲劳。

## 证实的原因：基础设施 Bug

### Anthropic 的 2025 年 9 月事后分析

2025 年 9 月，Anthropic 发布了一份详细的事后分析，揭示了**三个独立的基础设施 bug**，这些 bug 在 2025 年 8 月至 9 月期间降低了 Claude 的质量。他们的官方声明：

> "我们永远不会因为需求、一天中的时间或服务器负载而降低模型质量。用户报告的问题仅由基础设施 bug 导致。"

### Bug #1 — 上下文窗口路由错误

Sonnet 4 请求被意外路由到配置为 100 万令牌上下文窗口的服务器，而不是标准服务器。

- **时间线：** 2025 年 8 月 5 日引入，2025 年 8 月 29 日负载均衡变更后恶化
- **峰值影响：** 最严重时 16% 的 Sonnet 4 请求受影响（8 月 31 日）
- **用户影响：** 约 30% 的 Claude Code 用户至少收到一条质量下降的消息
- **隐蔽细节：** 路由是"粘性的"——一旦遇到坏服务器，后续请求会继续流向那里
- **修复：** 2025 年 9 月 4-18 日（跨平台推出）

### Bug #2 — TPU 输出损坏

TPU 服务器上的配置错误导致令牌生成期间出错，为本应很少出现的令牌分配高概率。

- **症状：** 英文响应中出现泰语或中文字符，明显的代码语法错误
- **受影响：** Opus 4.1 和 Opus 4（8 月 25-28 日），Sonnet 4（8 月 25 日-9 月 2 日）
- **范围：** 仅 Claude API；第三方平台不受影响
- **修复：** 9 月 2 日回滚

### Bug #3 — XLA:TPU 编译器错误编译（最棘手的）

一个修复精度问题的代码更改意外暴露了 Google XLA:TPU 中的**潜在编译器 bug**。

- **根本原因：** 近似 top-k 操作（用于选择最可能的下一个令牌）"有时会返回完全错误的结果，但仅针对某些批处理大小和模型配置"
- **为什么难以发现：** 它根据在它之前或之后运行的操作，以及是否启用了调试工具，会改变行为
- **隐藏数月：** 2024 年 12 月的先前变通方法意外掩盖了这个更深的 bug
- **受影响：** Haiku 3.5 已确认；Sonnet 4 和 Opus 3 的子集涉嫌
- **解决方案：** 从近似切换到精确 top-k；接受"轻微的效率影响"，因为"模型质量是不可协商的"

### 为什么检测困难

Anthropic 自己的自动评估没有捕获用户报告的退化，"部分原因是 Claude ���常能够很好地从孤立错误中恢复过来。"每个 bug 在不同平台上以不同速率产生不同症状，"形成了令人困惑的报告混合，没有指向任何单一原因。"

关键背景：Claude 运行在**三个不同的硬件平台**（AWS Trainium、NVIDIA GPU、Google TPU）上，每个都有不同的故障模式、编译器和精度行为。你的请求在不同日子可能 hit 不同的硬件。

## 证实的原因：MoE 路由差异

现代大型模型通常使用**专家混合（MoE）架构**，其中只有模型参数的子集（"专家"）针对每个输入激活。学习路由器决定使用哪些专家。

Scale AI 的研究揭示了一个关键发现：

> "稀疏 MoE 和批量推理的结合会产生不可预测的结果，因为批次的组成决定了你的查询被路由到哪个专家，而同一批次中其他用户查询的混合不是确定性的。"

### 跨提供商的日间差异测量

| 提供商 | 日间分数差异 |
|----------|--------------------------|
| OpenAI（GPT-4 变体）| ±10–12% |
| Anthropic（Claude 变体）| ±8–11% |
| Google（Gemini 变体）| ±9–14% |

具体例子：同一模型**一天在越狱抵抗上得分 77%，第二天得分 63%**。相同的模型、相同的权重、相同的测试——仅基础设施就产生了 14 个百分点的波动。

这意味着即使零 bug 和零变化，相同的模型也可能因请求的批处理和路由方式不同而在不同日子产生明显不同的质量输出。当日间噪声为 10-15% 时，A/B 测试无法可靠地检测 5% 的质量信号。

## 证实的原因：系统提示和后训练更新

### 系统提示更改

模型权重不变化，但包裹这些权重的**系统提示**可以随时更新。对 Claude 系统提示演变的分析显示了数十次迭代，"热修复"——添加修补不良行为的简短指令——被定期添加和删除。

Claude 3.7 的系统提示包含多个针对常见 LLM"陷阱"的热修复指令。Claude 4.0 的系统提示删除了所有这些指令，行为在 post-training 期间通过强化学习处理。

### 后训练理论

对于无法解释的质量变化，最合理的理论：公司在不更改基础模型权重的情况下可以更新**微调和 RLHF**（来自人类反馈的强化学习）。这在技术上可以说"模型没有变化"，同时仍然通过更新的安全护栏和指令遵循调整来改变行为。

## 证实的原因：静默模型切换

OpenAI 已被多次记录静默更改用户交互的模型：

- 一夜之间删除模型选择器，强制用户从 GPT-4o 切换到 GPT-5
- 将 GPT-4o 变成隐藏的"传统模型"，需要手动切换设置，但应用内没有通知
- 一个"自动切换器"bug 将用户路由到错误的模型
- Plus 订阅用户报告模型切换到"受限版本"未经同意

Sam Altman 承认推出"比我们希望的略显坎坷"。Reddit 帖子获得数千票，称新模型是"灾难"和"降级"。

这表明模型切换**在行业中确实会发生**——有时是故意的（产品决策），有时是意外的（路由 bug）。

## 影响因素

### 负载下的量化

为了经济高效地服务数百万用户，公司可能会服务模型的**量化**版本——将精度从 FP16 降至 INT8 或 INT4。这可以将内存使用减少 2-4 倍并加速推理，但会带来微妙的质量损失。提供商是否在负载下动态切换量化级别还有争议，但技术能力存在，在 vLLM 和 TensorRT 等服务框架中有充分文档记录。

### 投机解码

现代服务栈使用较小的"草稿"模型来预测多个令牌，然后由真正的模型验证它们。理论上这保持相同的输出分布，但实践中接受率因领域和上下文而异。开箱即用的草稿模型在某些情况下可能很好，但通常在领域特定任务或很长的上下文中会遇到困难。

### 上下文窗口污染

在长编码会话中，早期的错误在上下文中累积。模型看到自己的错误，可能会延续它们。这是会话内"Claude 变笨"的最常见原因——不是模型退化，而是上下文污染。

**实用技巧：** 当感觉质量不对时，使用 `/compact` 或开始新会话。这是你能做的最有操作性的事情。

## 斯坦福研究 — 为什么它很复杂

2023 年斯坦福和 UC Berkeley 的里程碑研究（Chen、Zaharia、Zou）——"ChatGPT 的行为如何随时间变化？"——经常被引用为 LLM 退化的证据。主要发现：

> GPT-4 在"Is this number prime? Think step by step"上的准确率从 2023 年 3 月的 **97.6%** 降至 2023 年 6 月的 **2.4%**。

### 研究证明了什么

- 相同 LLM 服务的**行为**可以在短时间内大幅变化
- 不同能力可能向相反方向移动（GPT-4 数学变差，GPT-3.5 变好）
- 代码生成质量下降（GPT-4 可执行代码：52% → 10%）
- 研究创造了**"LLM 漂移"**这个术语

### 方法论批评

- 3 月版本使用 **temperature 0.0**，而 6 月版本使用 **temperature 1.0**——一个增加随机性的基本混淆变量
- 每个任务只有 **500 个查询**——太小无法做出确定的统计声明
- "数学问题"实际是是/否问题，模型的猜测模式改变了，而不是其数学能力
- 变化可能反映了故意的**后训练安全更新**，而非退化

研究证明了重要的事情——LLM 行为随时间变化——但机制可能是故意的更新，而非非故意的退化。

## 心理因素

### 确认偏差

一旦有人发推"Claude 今天很笨"，你就会开始注意到每一个错误。在没有人抱怨的日子里，你会忽略同样的错误。社交媒体放大了这个效应。

### 蜜月效应

用户在新模型上经历初始蜜月期间，然后逐渐发现限制。模型没有变化——期望调整得比能力提升更快。

### 任务难度差异

你的任务日复一日地变化。��难问题多的日子会感觉模型变差了，即使它没有。

### "周末 Claude"神话

尽管许多用户相信一周中的日期模式，但严格分析**没有找到一周中质量模式的一致证据**。一项名为"AI 周一更笨"的分析无功而返。

### LLM 的本质

LLM 是概率性的。相同的提示每次可能产生不同的输出。在运气不好的情况下，你可能会连续获得几个较差的回复——纯随机性，而非退化。

## 底线

用户描述的现象**真实但归因错误**：

- **正确：** 他们在某些日子体验到质量下降
- **错误：** 模型被故意"削弱"

实际原因是以下因素的组合：

1. **基础设施 bug** — Anthropic 事后分析已证实（高达 16% 的请求受影响）
2. **MoE 路由差异** — Scale AI 测量到 ±8-14% 质量波动，即使零变化
3. **系统提示和后训练更新** — 各提供商有文档记录
4. **硬件异构性** — TPU vs GPU vs Trainium，每个有不同的故障模式
5. **上下文污染** — 长会话在会话内质量下降
6. **确认偏差** — 社交媒体放大了感知到的模式
7. **本质差异** — 相同的模型、相同的提示，每次输出不同

测量问题很严重：日间差异 ±8-14% 意味着你无法将真实的 5% 质量变化与噪声区分开来。这就是为什么"这都在你脑海里"和"他们削弱了它"两派都自信——信噪比使得仅凭个人体验无法判断。

# 🔥 Codex 5.3 高推理和发现

### 报告范围

本节解释为什么用户可能会在短时间内体验到 Claude 输出质量下降，而 Codex 5.3 在编码任务上感觉稳定或更强。重点不是永久模型质量排名。重点是真实服务条件下的短期生产行为。

报告日期：2025 年 3 月 5 日。

### 观察到的模式

报告的模式是：

1. 一段时间内质量可接受。
2. ���量���乎下降了几天。
3. 质量接近之前的基线恢复。

这种形状通常是服务栈或推出模式，而非永久性的基础模型能力变化。永久性的能力下降通常不会在没有明确回滚或修复的情况下这么快恢复。

### 高推理：为什么 Codex 5.3 在糟糕的窗口中看起来更好

Codex 5.3 在另一个提供商的降级期间可能看起来明显更强，原因有多种技术原因，可以同时发生：

1. 产品目标匹配。Codex 5.3 针对代码生成和代理编码工作流进行了优化，因此即使原始模型力量相等，由于工具编排、仓库推理和以代码为中心的指令微调，也能产生更好的编码结果。
2. 推理策略差异。各提供商独立调整延迟、推理深度和解码默认值。一个提供商更保守的策略在同一天可能看起来比另一个提供商激进的速度优化策略"更聪明"。
3. 服务路径分离。即使两个提供商托管最先进模型，它们运行不同的路由层、编译器/运行时栈和推出流水线。一个栈中的事故并不意味着另一个相关联地退化。
4. 推出和回滚时机。如果一个提供商正在推出而另一个稳定，用户可能会看到大的临时质量差异，而底层模型权重没有长期变化。
5. 会话级污染效应。在长编码聊天中，错误累积可以放大感知到的下降。一个竞争对手助手可能感觉更好，仅仅因为失败的会话被重置，或者因为它的工具循环恢复得更快。

### 详细发现

对于"Claude 感觉非常弱大约四天，然后恢复了"这样的报告，最可能的解释是：

1. 提供商端的事故、路由问题、解码/运行时 bug 或推出回归影响了请求的子集。
2. 问题持续足够长以在真实工作流中反复被注意到。
3. 问题被修复或回滚。
4. 感知到的质量快速恢复。

在同一期间，Codex 5.3 可能感觉明显更好，因为它没有共享相同的事故路径，而且编码任务优化放大了实践结果中的差距。

### 此模式的假设排名

| 假设 | 可能性 | 理由 |
|------------|------------|-----------|
| 提供商事故加回滚 | 高 | 最匹配多天骤降然后快速恢复 |
| 服务配置更改（采样/延迟/推理预算）| 高 | 行为突然变化而无模型再训练的常见来源 |
| 静默别名或快照移动 | 中高 | 可以在没有可见用户操作的情况下改变行为 |
| 仅提示漂移和上下文污染 | 中 | 可以降解会话，但单独不太可能解释广泛的多天报告 |
| 永久基础模型退化 | 低 | 与快速恢复到之前质量不一致 |

### 什么会证实或否定这一发现

要将其从高置信度推理变成硬证据，收集跨天的相同任务集的请求级遥测：

1. 请求时的确切模型标识和快照/别名。
2. 提供商公开的任何后端指纹或发布标记。
3. 解码参数（temperature、top_p、top_k、max tokens）。
4. 延迟、超时和错误率跟踪。
5. 固定编码基准提示集上的结构化质量分数。
6. 失败点的会话长度和令牌上下文深度。

如果质量下降与事故窗口、配置更改或后端指纹变化相关联，则事故/配置假设被证实。如果不存在这样的变化，而退化仅在长会话中，则上下文污染成为主要解释。

### 实际工程建议

为了减少生产中的日间差异：

1. 在可用时固定模型快照，而不是使用浮动别名。
2. 存储请求元数据（模型 ID、参数、延迟、错误、响应质量标签）。
3. 为编码任务运行固定的每日金丝雀套件并在回归时警报。
4. 几次失败轮次后重置或压缩长运行会话。
5. 为事故窗口保留备用提供商/模型路径。
6. 在内部仪表板中将"模型质量"与"服务可靠性"分开。
