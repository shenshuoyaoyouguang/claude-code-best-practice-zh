description: Execute phased implementation with validation gates
argument-hint: "<feature-slug> [--phase N] [--validate-only]"

## 用户输入

```text
$ARGUMENTS
```

您必须解析用户输入以提取功能 slug（rpi/ 中的文件夹名称）。

## 目的

此命令根据规划文档执行分阶段实施。它编排专门的 agent，强制执行验证门，并在整个实施过程中确保宪法合规。

**前提条件**：
- 功能文件夹位于 rpi/{feature-slug}/
- 规划完成（rpi/{feature-slug}/plan/PLAN.md 存在）

**输出位置**：rpi/{feature-slug}/implement/

**这是 RPI 工作流的第 4 步**（最后一步 - 实际实施）。

## 标志

- --phase N：执行特定阶段号（1-8），如果省略从阶段 1 开始
- --validate-only：仅验证当前阶段，不实施
- --skip-validation：跳过验证门并继续（谨慎使用）

## 可用 Agent

所有 agent 都使用 Opus 模型以获得最大质量。

### 实施 Agent

| Agent | 类型 | 使用时机 |
|-------|------|-------------|
| senior-software-engineer | 自定义 | 所有实施任务 |

### 支持 Agent

| Agent | 类型 | 目的 |
|-------|------|---------|
| Explore | 内置 | 实施前的代码探索 |
| code-reviewer | 自定义 | 代码审查和质量验证 |
| constitutional-validator | 自定义 | 针对项目宪法进行验证 |
| documentation-analyst-writer | 内置 | 文档生成 |

### Agent 路由

所有实施任务由 senior-software-engineer agent 处理。

## 阶段 0：加载上下文和规则

**前提条件**：从用户输入解析功能 slug

**流程**：

### 0.1 加载项目宪法

1. 检查仓库中是否存在宪法或原则文档
2. 如果存在，提取：
   - 技术约束（类型安全、测试、组件隔离）
   - 业务原则（质量标准、工作流）
   - 架构边界
3. 在实施期间存储约束以强制执行

### 0.2 加载领域特定指南

基于要修改的文件，加载相关的项目指南：
- 检查特定组件的 README 文件
- 检查编码风格指南
- 检查测试要求文档

### 0.3 分析实施范围

1. 读取 rpi/{feature-slug}/plan/PLAN.md
2. 识别所有要修改的文件
3. 将文件映射到实施 agent

**输出**：
- 宪法上下文摘要
- 已加载的领域规则
- 文件到 agent 映射
- 阶段执行计划

**验证**：
- [ ] 宪法已加载（如果存在）
- [ ] 已为受影响的文件加载领域规则
- [ ] 所有文件已映射到 agent
- [ ] 执行计划已理解

## 分阶段实施工作流

### 阶段实施循环

对于 PLAN.md 中的每个阶段：

```
┌─────────────────────────────────────────────────────────────────┐
│ 阶段 N: [阶段名称]                                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1. 代码发现（Explore Agent）                                    │
│     └─→ 更改前了解现有代码                                       │
│                                                                  │
│  2. 实施（senior-software-engineer）                              │
│     └─→ 实现阶段交付物                                           │
│                                                                  │
│  3. 自我验证                                                     │
│     └─→ 工程师根据阶段清单进行验证                                │
│                                                                  │
│  4. 代码审查（code-reviewer Agent）                              │
│     └─→ 安全性、正确性、可维护性                                  │
│                                                                  │
│  5. 用户验证门                                                   │
│     └─→ 停止并请求用户批准                                        │
│         ├─→ 通过：继续下一阶段                                    │
│        ├─→ 条件通过：记录问题，继续                               │
│        └─→ 失败：修复问题，重新验证                               │
│                                                                  │
│  6. 文档更新                                                     │
│     └─→ 在 PLAN.md 中更新阶段状态                                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 步骤 1：代码发现（每个阶段）

**Agent**：Explore（通过 Task 工具，内置）

**目的**：在更改之前将实施扎根于代码现实。

**流程**：
1. 通过 Task 工具启动 Explore agent，subagent_type="Explore"
2. 请求分析当前阶段受影响的文件
3. 了解现有模式、集成点、约束

**Explore Agent 提示**：
```
分析代码库以准备实施 [feature-name] 的阶段 N。

本阶段要修改的文件：
[从 PLAN.md 列出文件]

调查并记录：

1. 当前实现
   - 这些文件当前如何工作？
   - 使用什么模式？
   - 哪些函数/类会受影响？

2. 集成点
   - 其他哪些文件导入或使用这些模块？
   - 哪些 API 或接口会改变？
   - 哪些测试覆盖这段代码？

3. 依赖
   - 使用哪些库？
   - 有哪些内部工具可用？
   - 现有代码有什么约束？

4. 要遵循的模式
   - 这些文件使用什么编码风格？
   - 遵循什么命名约定？
   - 存在什么错误处理模式？

5. 风险和注意事项
   - 如果我们更改这个会破坏什么？
   - 存在什么边缘情况？
   - 有什么向后兼容性顾虑？

提供发现摘要以指导实施。
```

**输出**：实施 agent 的发现摘要

## 步骤 2：实施（每个阶段）

**Agent**：senior-software-engineer

**流程**：
1. 使用 senior-software-engineer agent
2. 从步骤 1 提供发现上下文
3. 实现阶段的所有交付物
4. 遵循宪法约束和项目规则

**实施 Agent 提示模板**：
```
作为 [agent-name] agent，实施 [feature-name] 的阶段 N 交付物。

## 上下文
- 宪法约束：[从阶段 0]
- 领域规则：[从阶段 0]
- 发现摘要：[从步骤 1]

## 阶段 N 交付物
[从 PLAN.md 列出]

## 要修改的文件
[从 PLAN.md 列出具体更改]

## 实施要求
1. 遵循在发现中识别的现有代码模式
2. 遵守宪法约束（类型安全、测试等）
3. 遵循项目特定规则（如适用）
4. 为新功能编写测试
5. 包含适当的日志记录
6. 优雅地处理错误

## 质量清单
- [ ] 代码遵循现有模式
- [ ] 适用处存在类型注解
- [ ] 测试编写并通过
- [ ] 对现有功能没有破坏性更改
- [ ] 添加了可观察性的日志
- [ ] 错误处理全面

实施所有交付物并报告完成情况。
```

## 步骤 3：自我验证

**Agent**：senior-software-engineer（与步骤 2 相同）

**流程**：
1. agent 根据阶段清单验证实施
2. 运行 linting（使用项目配置的 linter）
3. 运行与更改相关的测试
4. 验证构建成功

**验证命令**（根据您的项目调整）：

```bash
# 运行 linter
[your-linter-command]

# 运行测试
[your-test-command]

# 构建/编译
[your-build-command]
```

**自我验证清单**：
- [ ] 所有交付物已实施
- [ ] Linting 通过
- [ ] 测试通过
- [ ] 构建成功
- [ ] 现有测试没有回归
- [ ] 遵守宪法约束
- [ ] 遵循领域规则

## 步骤 4：代码审查

**Agent**：code-reviewer（自定义，自动调用）

**流程**：
1. 调用 code-reviewer agent 审查更改
2. 关注正确性、安全性、可维护性
3. 在继续之前解决阻塞问题

**代码审查 Agent 提示**：
```
作为 code-reviewer agent，审查 [feature-name] 的阶段 N 实施。

## 修改的文件
[列出修改的文件]

## 所做更改
[实施摘要]

## 审查重点
- 正确性和测试
- 安全性和依赖卫生
- 架构边界
- 清晰优先于巧妙

## 宪法约束
[从阶段 0]

使用标准输出格式提供审查。
```

**审查裁决**：
- **批准**：继续进行用户验证
- **批准但有建议**：记录建议，继续
- **需要修改**：修复问题，重新审查

## 步骤 5：用户验证门

**关键**：此步骤需要用户交互。不要自动继续。

**流程**：
1. 呈现阶段交付物清单
2. 显示已实施的内容（修改的文件、添加的功能）
3. 显示来自 PLAN.md 的验证标准
4. 显示代码审查结果
5. **停止并等待用户决定**

**验证请求格式**：
```
## 阶段 N 验证请求

### 已完成交付物
- [x] [交付物 1] - [实施摘要]
- [x] [交付物 2] - [实施摘要]
- ...

### 修改的文件
| 文件 | 更改类型 | 行数 |
|------|-------------|-------|
| [file] | [add/modify] | [±N] |

### 测试
- [x] 单元测试：通过
- [x] 集成测试：通过
- [x] 构建：成功

### 代码审查
- 裁决：[批准 / 批准但有建议]
- 问题：[无 / 列出]

### 验证标准（来自 PLAN.md）
- [ ] [标准 1]
- [ ] [标准 2]
- ...

---

**请验证阶段 N**：
- **通过**：阶段完成，继续阶段 N+1
- **条件通过**：在下面记录问题，谨慎继续
- **失败**：指定在继续之前需要修复的问题
```

**用户决定**：
- **通过**：继续下一阶段
- **条件通过**：记录问题，继续下一阶段
- **失败**：修复问题，重新运行步骤 2-5

**流程**：

3. 将每个阶段的验证追加到 IMPLEMENT.md

## 错误处理

### 实施失败

**如果实施失败**：
1. 记录具体失败
2. 分析根本原因
3. 尝试替代方法（最多 2 次）
4. 如果仍然失败，停止并请求用户指导
5. 不要继续进行下一阶段，同时实施已损坏

**消息**："实施失败：[error]。已尝试 N 种方法。需要用户指导。"

### 测试失败

**如果测试失败**：
1. 分析失败原因（代码 bug vs 测试 bug）
2. 修复问题
3. 重新运行测试
4. 如果持续存在，记录并询问用户
5. 不要将阶段标记为完成，同时测试失败

**消息**："测试失败：[failures]。已尝试修复但不成功。需要用户审查。"

### 构建失败

**如果构建失败**：
1. 检查类型错误
2. 检查缺失的导入
3. 检查语法错误
4. 修复并重新构建
5. 如果持续存在，升级到用户

**消息**："构建失败：[error]。无法自动解决。"

### Agent 失败

**如果 agent 失败或超时**：
1. 使用相同输入重试一次
2. 如果仍然失败，在没有该 agent 贡献的情况下继续
3. 在验证请求中记录差距

**消息**："Agent [name] 失败。在没有贡献的情况下继续。"

## 备注

### 何时使用此命令

- 在 /rpi:plan 生成 PLAN.md 之后
- 当需要带验证门的分阶段实施时
- 对于需要结构化实施的功能

### 何时不使用此命令

- Bug 修复（太重，直接修复即可）
- 非常简单的更改（<30 分钟工作）
- 探索性原型
- 仅文档更改

### 最佳实践

1. **首先查看 PLAN.md**：了解您要实施的内容
2. **信任代码发现**：让 Explore agent 指导实施
3. **遵循现有模式**：让代码发现指导实施
4. **不要跳过验证**：门存在是为了尽早发现问题
5. **边做边记录**：每个阶段后更新状态
6. **卡住时询问**：询问总比错误地继续好

### RPI 工作流的一部分

4 步中的第 4 步（描述 → 研究 → 计划 → 实施）

## 完成后操作

**重要**：在完成实施（所有阶段或重大进展）后，始终提示用户压缩对话：

> **上下文管理**：此实施工作流消耗了大量上下文。为了保留进度并释放空间，请运行：
>
> ```
> /compact
> ```
>
> 这将总结对话并保留实施状态，同时减少未来工作的 token 使用。

**何时提示压缩**：
- 所有阶段完成后
- 如果是多会话实施，则在每个主要阶段完成后
- 如果在实施期间上下文不足
